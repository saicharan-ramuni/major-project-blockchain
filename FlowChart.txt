Blockchain-Based anonymous authentication framework for Healthcare IoT Workflow (Flask + Web3 + AES + SMTP)
===========================================================

---------------------------------------------------
1. SYSTEM INITIALIZATION
---------------------------------------------------
1. Start Ganache/local Ethereum on 127.0.0.1:8545
2. Deploy Report.sol using Truffle
3. Start CSP socket server (run CSP.py) to receive tx receipts
4. Launch Flask app (app.py)
5. Flask loads contract ABI from Report.json
6. Flask uses deployed contract address in app.py
7. AES-256-GCM key loaded from app.py constant
8. SMTP credentials loaded from mailer.py
9. System ready

---------------------------------------------------
2. ADMIN LOGIN
---------------------------------------------------
1. Admin opens /AdminLogin
2. Enters admin/admin
3. If valid -> Admin dashboard

---------------------------------------------------
3. ADD DOCTOR / HOSPITAL (ADMIN)
---------------------------------------------------
1. Admin opens /AddDoctor
2. Enters:
      - Doctor username, password
      - Email, contact, qualification, experience
      - Hospital/Doctor name
      - Address
3. System checks blockchain hospital records
4. If username exists -> reject
5. Else:
      - Create record: hospital#...#email#...#hospitalName
      - Encrypt line (AES-256-GCM) and append on-chain
6. Doctor/hospital stored on-chain

---------------------------------------------------
4. PATIENT SIGNUP
---------------------------------------------------
1. Patient opens /PatientSignup
2. Enters:
      - Username, password
      - Email, contact, address
3. System checks signup records on-chain
4. If username exists -> reject
5. Else:
      - Create record: signup#user#pass#contact#email#address
      - Encrypt line and append on-chain
6. Signup stored on-chain

---------------------------------------------------
5. PATIENT LOGIN
---------------------------------------------------
1. Patient opens /PatientLogin
2. Enters username + password
3. System decrypts and checks signup records
4. If valid -> Patient dashboard

---------------------------------------------------
6. PATIENT SHARE REPORT
---------------------------------------------------
1. Patient opens /AddHealth
2. Enters:
      - Age
      - Symptoms
      - Upload report file
      - Select hospital/doctor(s)
3. System encrypts report file with AES-256-GCM
4. Encrypted file saved as static/reports/<filename>.enc
5. Patient record created:
      patient#user#age#symptoms#hospitals#filename#date
6. Record encrypted and appended on-chain
7. SMTP mail sent to selected hospital emails
      - Includes patient name, symptoms, date
      - Report attached

---------------------------------------------------
7. DOCTOR LOGIN
---------------------------------------------------
1. Doctor opens /DoctorLogin
2. Enters username + password
3. System decrypts hospital records and validates
4. If valid -> Doctor dashboard

---------------------------------------------------
8. DOCTOR VIEW REPORTS
---------------------------------------------------
1. Doctor opens /ViewPatientReport
2. System decrypts patient records
3. Shows only records shared to doctor’s hospital
4. Each row shows:
      - Patient details
      - Encrypted report download link (served decrypted)
      - Prescription status

---------------------------------------------------
9. DOCTOR PRESCRIPTION
---------------------------------------------------
1. Doctor clicks Prescription for a report
2. Enters prescription text
3. Prescription record created:
      prescription#pname#pdate#filename#prescription#doctor#date
4. Record encrypted and appended on-chain
5. Patient email is looked up from signup record
6. SMTP mail sent to patient with:
      - Patient name, report date, prescription text
      - Report attached (decrypted)

---------------------------------------------------
10. PATIENT VIEW / DOWNLOAD
---------------------------------------------------
1. Patient opens /ViewHealth
2. System decrypts patient records
3. Shows only patient’s own reports
4. Download uses /download_report?name=<filename>
5. Server decrypts file from .enc and sends to browser

---------------------------------------------------
11. CSP RECEIPTS
---------------------------------------------------
1. Flask sends tx receipt to CSP via socket 2222
2. CSP prints receipts in terminal

---------------------------------------------------
12. SYSTEM TERMINATION
---------------------------------------------------
1. Stop Flask app
2. Stop CSP server
3. Stop Ganache
4. On-chain data remains in blockchain
